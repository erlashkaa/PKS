# Тестирование симулятора сетей

Система тестирования симулятора состоит из двух независимых уровней: **Юнит-тестов (Unit Tests)** для проверки корректности отдельных классов и **Интеграционных сценариев (Scenarios)** для тестирования всей топологии в сборе.

---

## 1. Юнит-тесты (Unit Tests)
Файл: `tests/unit_tests.cpp`

Юнит-тесты проверяют изолированную логику и математику каждого базового класса симулятора без запуска глобальной сети:

* **`PacketTest`**: Проверяет конструктор, корректность заполнения IP/MAC адресов и автоматическую генерацию глобального уникального ID для каждого нового пакета.
* **`HostTest`**: Проверяет работу геттеров имени/адресов, инициализацию счетчиков (sent, received, dropped), а также логику избирательного приема unicast-пакетов (игнорирование чужого трафика).
* **`RouterTest`**: Тестирует статические таблицы маршрутизации, сопоставление IP-сетей по маске подсети и пересылку пакетов между изолированными сегментами L2.
* **`CSMACDMediumTest`**: Проверяет физическую математику времени передачи (время сериализации + задержка распространения сигнала) и управление состоянием занятости среды (Carrier Sense / isIdle).
* **`StatisticsCollectorTest`**: Валидирует корректность отслеживания событий среды (PACKET_SENT, PACKET_DELIVERED, COLLISION_DETECTED) и работу функции сброса (`reset()`).

---

## 2. Интеграционные сценарии (Scenarios)
Сценарии проверяют работу всей сети в сборе. Топология и параметры настраиваются через фасад (`SimulationFacade`).

**`Scenario1_SimpleTransfer` (Файл: `scenario1_simple_transfer.cpp`)**
* **Суть**: Простейшая сеть. 2 хоста напрямую соединены одной средой передачи (LAN).
* **Что тестирует**: Базовую доставку. Поскольку хостов всего два и отправляет только один, коллизий быть не должно. Проверяет, что доля успешной доставки (Delivery Rate) близка к 100%.

**`Scenario2_Collision` (Файл: `scenario2_collision.cpp`)**
* **Суть**: Сеть с высокой конкуренцией. Несколько хостов спамят пакетами в общую шину одновременно.
* **Что тестирует**: Работу алгоритма CSMA/CD. Сценарий подтверждает, что при одновременной отправке фиксируются **коллизии**. Также тест проверяет, что благодаря алгоритму `Backoff` (задержки при повторной передаче) узлы все равно ухитряются протолкнуть большую часть пакетов, а не блокируют сеть намертво.

**`Scenario3_Routing` (Файл: `scenario3_routing.cpp`)**
* **Суть**: Сложная топология с двумя подсетями.
  * Host-A в LAN1 (192.168.1.0/24)
  * Host-B в LAN2 (192.168.2.0/24)
  * Между ними Router, соединенный с обоими LAN.
* **Что тестирует**: Интеграцию всех слоев сразу. Пакет от Host-A должен уйти в среду LAN1, быть перехвачен маршрутизатором, роутер должен сопоставить IP-адрес с таблицей маршрутизации, перекинуть пакет в интерфейс LAN2, откуда среда LAN2 доставит его на конечный Host-B. Тест запрашивает у Host-B статистику принятых пакетов и подтверждает успех.

**`Scenario4_HeavyLoad` (Файл: `scenario4_heavy_load.cpp`)**
* **Суть**: Стресс-тест сети с экстремально короткими таймингами отступа (backoff). 
* **Что тестирует**: Сильную нагрузку на общую шину, где несколько узлов агрессивно отправляют `Bursty` (пульсирующий) трафик в одну среду передачи с малым `backoffSlot` и малым `maxRetries`. Ожидается огромное количество коллизий и отбрасываний пакетов, проверяется устойчивость симулятора под нагрузкой.

**`Scenario5_MixedTraffic` (Файл: `scenario5_mixed_traffic.cpp`)**
* **Суть**: Смешение различных паттернов генерации трафика в одной сети.
* **Что тестирует**: Узел с постоянной скоростью `CBR` и узел со случайным `Poisson` распределением пытаются отправить пакеты на один и тот же целевой узел. Гарантирует, что обе стратегии корректно работают вместе, и целевой хост успешно получает пакеты от обоих отправителей.

**`Scenario6_NetworkFailure` (Файл: `scenario6_network_failure.cpp`)**
* **Суть**: Отправка пакетов в несуществующую сеть (или без настроенного маршрута).
* **Что тестирует**: Маршрутизатор подключен к сети, но у него пустая таблица маршрутизации. Хост отправляет пакет на адрес `172.16.x.x`, который доходит до маршрутизатора. Роутер, не зная куда отправить пакет дальше, обязан безопасно его **отбросить** (drop), не вызывая исключений и падений.

## Структура файлов

```
project/
├── run_tests.bat                        # Главный скрипт (сборка и запуск сценариев)
├── TESTING.md                           # Этот файл с документацией
└── tests/
    ├── CMakeLists.txt                   # CMake-конфигурация тестов
    ├── unit_tests.cpp                   # Юнит-тесты кор-классов
    └── scenarios/                       # Интеграционные тесты-сценарии
        ├── scenario1_simple_transfer.cpp # Простая передача без коллизий
        ├── scenario2_collision.cpp       # Коллизии и backoff
        ├── scenario3_routing.cpp         # Маршрутизация между подсетями
        ├── scenario4_heavy_load.cpp      # Стресс-тест с отбрасываниями
        ├── scenario5_mixed_traffic.cpp   # Разные паттерны (Poisson + CBR)
        └── scenario6_network_failure.cpp # Проверка обрыва маршрутов
```

## Категории сценариев (всего 6)

### Базовая топология
- Scenario 1: Простая передача между хостами без коллизий
- Scenario 2: Обнаружение коллизий, алгоритм backoff, повторная передача

### Маршрутизация
- Scenario 3: Маршрутизация между подсетями через статическую таблицу маршрутов
- Scenario 6: Безопасное отбрасывание пакетов при отсутствии нужного маршрута

### Нагрузка и трафик
- Scenario 4: Высокая конкуренция, исчерпание лимита повторных передач (Drops)
- Scenario 5: Комбинированная нагрузка (Poisson + CBR) на один получатель

## Добавление новых сценариев

При добавлении нового сценария:

1. Создайте его в папке `tests/scenarios/`
2. Подключите в `tests/CMakeLists.txt` через макрос `add_scenario(...)`
3. Опишите в данном файле (`TESTING.md`).

## Командная строка

```powershell
# Запуск всех сценариев (конфигурация + сборка + тест)
.\run_tests.bat

# Запуск конкретного сценария (при уже собранном проекте)
cd build
ctest -R "Scenario4" --output-on-failure
```
